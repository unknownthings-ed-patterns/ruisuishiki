# 背骨設計ハンドオフ — Fable 不在時に Opus が背骨を設計するための条件と手順

作成: Fable 5（2026-07-02）。

**経緯**：単元の系列構造の設計（背骨）は本来 Fable の担当（資料共有 → Fable 背骨設計 → Opus 実装 → レビュー → 先生が歩いて磨く）。しかし 2026-07-01 に Fable 5 が利用不可になった期間、実装は「メイン（Opus 4.8）が直接実装」のフォールバックで乗り切れたが、**背骨設計のフォールバック手順は無かった**。本文書は、既済の背骨 2 本——`docs/trig_series_design_fable.md`（系列1〜3・2026-06-27）と `docs/trig_continuation_series_design_fable.md`（系列4〜9・2026-07-02）——を**比較して**背骨設計の条件を抽出し、Fable 不在時に **Opus が背骨ドラフト → 先生検証のループ**で進められるようにする手順書である。

（比較から基準を抽出するのは、お手本 7 条件を 2 系列の比較で出したのと同じ方法＝**比較が基準を生む**、推理式の方法のメタ適用。）

---

## 1. 背骨とは何か（定義）

背骨＝系列を実装する**前**に確定させる設計文書。1 系列につき次の 6 点：

1. **中心の問い**（胚細胞基準・1〜2 文）＋その胚細胞的概念と転移先
2. **10 ステップ表**（各 step の内容を**構造で**記述・op 列・答えの型/affordances 列・図列）
3. **オペレータ配分**（5 つ全部・一度に差異 1 つ）
4. **既知→未知の橋**（step1 Socratic の案・指す前系列/他ユニット既習・`[用語]`）
5. **縦の鎖・横の橋**（前後の系列との接続・relatedSeriesId の向き）
6. **図と入力 affordances の指定**（流用/新規・新規図の内容一覧）

背骨に**含めないもの**：問題の具体値（実装時にオリジナルで起こす）・ヒント 3 層の文面・derivation 本文・コード。**具体値を含まないから docs/ に置いて公開リポジトリに commit できる**——これが消化メモ（vault・非公開）との置き場の違い。

## 2. 比較から抽出した「背骨設計の条件」C1〜C10

2 本の背骨（＋その間に起きた入力系ブロッカーの実体験）を比較して立ち上がった不変項と改善項：

| # | 条件 | どこから来たか |
|---|---|---|
| **C1** | **入力系ゲートを最初に通す**。全系列の「答えの型」を列挙し、現行入力系（数値・分数・`sqrt`・`pi`・`multi`・`answerDisplay`）で採点できるか表にする。できない答えがあれば、(a) **数値で問える形に設計を折る**（例：一般解→範囲付き有限解、グラフ→数値読み取り、合成の式→ r と α に分解）か、(b) 折れないなら**入力系拡張を先に**やる（系列2/3 の前例）。原則は「**採点できないものを無理に採点しない・概念は derivation で言葉として扱う**」 | 1 本目の背骨には無く、実装後に π/√ ブロッカーが発覚して入力系拡張を挟んだ。2 本目は冒頭 D2/D3 で先に確定。**最大の教訓** |
| **C2** | **中心の問いは胚細胞基準**（memory `feedback_driving_question_embryo_cell.md` の checklist）。さらに**ユニットのハブ胚細胞**（三角なら「単位円上の 1 点」）に各系列の問いを接続する。素材（消化メモ）の胚細胞候補は**種**であって最終形ではない——1〜2 文に磨く | 両背骨の不変項。2 本目で「ハブへの接続」を明示化 |
| **C3** | **10 step・5 オペレータ網羅・一度に差異 1 つ**。質的変化は後半の山場に置く。**逆と＋α は意識的に**（抜けやすい癖）。op 列を表に必ず書き、設計段階で 5 つ揃っているか数える | 両背骨の不変項（`ruisuishiki-core-principles`） |
| **C4** | **step1 は既知→未知の橋**。Socratic の問いの案を背骨に書く。指す先は「一つ前の系列」または「他ユニットの既習」（例：円の方程式・対称移動・二次関数の平行移動）。鍵概念は `[用語]` リンク（前提未実装なら素通し degrade で置いてよい） | 両背骨の不変項。2 本目で他ユニットへの横の橋を各系列に明記 |
| **C5** | **縦の鎖・横の橋を図式で書く**。relatedSeriesId は概念の**最も下**（最初に教わる系列）へ。後続系列が前の系列を**呼び出す** step（例：加法定理 step8 が相互関係を使う）も背骨で指定 | 両背骨の不変項 |
| **C6** | **図は step1＋質的変化 step のみ・答えを書かない**。背骨の段階で「既存マーカー流用か新規 SVG か」を**一覧表**にする（新規図の内容・使う step・「図に書いてはいけない値」まで）。実装時の手戻り予防 | 1 本目は「図：step1 と step4」程度の粒度。2 本目で新規/流用一覧（§8）に発展——グラフ系は流用元が無かったため |
| **C7** | **step 表に「答えの型 / affordances」列を持つ**（プレーン・分数・`[sqrt]`・`[pi]`・`[pi, multi]`）。C1 のゲートを step 粒度まで降ろしたもの。`answerDisplay` が要る値（π・無理数）もここで見える | 2 本目で新設。入力系ブロッカーの再発予防 |
| **C8** | **具体値を書かない**。step 内容は「第 2 象限で sin が与えられ cos を求める」のように**構造で**書く。値は実装時にオリジナル＋SymPy 検証（copyright-credit-vs-copy）。これで背骨を公開 repo に置ける | 1 本目は池田の練習問題の値を含んでいた（vault 外に置くには要注意だった）。2 本目で原則化 |
| **C9** | **系列の粒度（吸収 / 独立）の判断基準**：その内容**単独で** 5 オペレータ・10 step の厚みを満たせるか。満たせない薄さなら親系列に吸収（前例：周期→系列1）。満たせる厚みなら独立（前例：2倍角・半角）。迷ったら独立にして「吸収へのフォールバック（どの step をどう畳むか）」を背骨に書いておき、先生の歩行フィードバックで確定 | 1 本目の吸収前例＋2 本目の独立判断の比較から |
| **C10** | **順序は池田本の実際の節順**（先生が紙と並行して歩けることが最優先）。消化メモ・素材と本の順序が食い違ったら**本と先生が正**——背骨の冒頭に「確定事項」節を置き、素材からの変更点と番号対応を明記する | 2 本目で実際に発生（消化メモ「相互関係→性質→グラフ」→ 本は「相互関係→グラフ→性質」）。先生の指摘で確定 |
| **C11** | **質的変化 step は予想が裏切られる具体値・場面を選ぶ**（前 step までのやり方がそのまま通ると思わせてから、通らない） | 質的変化を「ただの難化」ではなく、比較から構造の再発見が起きる山場として設計するため |

## 3. Opus 用・背骨設計の手順（Fable 不在時のフォールバック）

### Phase 0 — 読む（着手前・必ず全部）

1. memory `feedback_ruisuishiki_core_principles.md`（5 オペレータ・比較の指さし・F1〜F10・進捗の正）
2. memory `feedback_driving_question_embryo_cell.md`（中心の問いの checklist）
3. memory `feedback_session_start_audit.md` の表にある新規系列向けメモリ一式
4. 本文書（C1〜C10）＋既済背骨 2 本（形式の mirror 元）
5. 先生が共有した素材（消化メモ等）。**素材に「未照合」等の正直な注記があれば背骨にも引き継ぐ**

### Phase 1 — 入力系ゲート（C1・最初に）

- 素材から各系列の「答えの型」を洗い出し、採点可否の表を作る
- 採点できない型があれば：数値で問える設計に折れるか検討 → 折れないなら**背骨より先に入力系拡張を先生に提案**（勝手に実装しない。系列2/3 のときの順序）
- ここで確定した方針は背骨冒頭の「確定事項 D1, D2, …」に書く

### Phase 2 — 粒度と順序（C9・C10）

- 系列の切り方（吸収/独立）を厚み基準で判断し、根拠を書く
- 順序は池田本の実際の節順。**素材と本の順序が食い違っている可能性があれば、背骨を書く前に先生に節順を確認する**（1 問だけの確認で済む）

### Phase 3 — 各系列の背骨を書く（C2〜C8）

§4 のテンプレで 1 系列ずつ。書きながら：

- 中心の問い：素材の胚細胞候補 →「エッセンスを 1 文で言うと？」→ 問いの形へ。タイトルの言い換え・操作の列挙になっていないか胚細胞 checklist で照合
- 10 step 表：op 列を先に設計（同・逆・＋α・質・複合の配置）→ 内容を構造で書く → affordances 列 → 図列
- 図：新規/流用一覧を系列横断でまとめる（Math.tsx の既存マーカーを grep して流用可能性を先に確認）

### Phase 4 — 自己監査（コミット前）

- [ ] 各系列：5 オペレータすべて最低 1 問（特に**逆**・**＋α**）
- [ ] 各系列：一度に差異 1 つ（step 間で二段飛ばしがない）
- [ ] 中心の問い：胚細胞 checklist 5 項目（原理を問う・転移が利く・steps が問いに答える 等）
- [ ] 全 step：答えが現行入力系で採点できる（できないものは設計を折ったか、拡張提案を先出ししたか）
- [ ] 図：step1＋質的変化のみ・「図に書いてはいけない値」を指定した
- [ ] 具体値ゼロ（公開 repo に置ける状態）
- [ ] 縦の鎖・横の橋・relatedSeriesId の向き・辞書候補を書いた
- [ ] 出典方針（APA＋「値はオリジナル」注記）を実装事項に書いた

### Phase 5 — 先生検証ループ

1. 背骨ドラフトを commit・push し、先生に**検証の観点を 3 つ以内**で添えて渡す。観点は毎回：
   - **順序と粒度**：紙（池田本）と並行して歩ける切り方・並びか（C9/C10 の判断は先生の本の実物が最終裁定）
   - **中心の問い**：単元の胚細胞を撃ち抜いているか（ラフでよい・歩いて磨く前提）
   - **入力方針**：設計を折った箇所（例：値で問う・分解して問う）が学びとして許容範囲か
2. フィードバックは**機械的に反映せず、違和感の根本を一緒に考える**（development-loop）。背骨を改訂して再 push
3. 先生の OK が出たら背骨を**凍結**し、実装フェーズへ（実装の作法は既済背骨の「実装上の必須事項」節と `hint_rewrite_handoff.md` に従う。**背骨フェーズでは実装しない**——フェーズを混ぜない）
4. 実装後、先生が歩いて出た違和感のうち**背骨レベルのもの**（順序・粒度・問いの狙い）は背骨文書に還流して改訂履歴を残す

### Phase 6 — 記録

- memory `feedback_ruisuishiki_core_principles.md` の三角（該当ユニット）進捗節に 1 行追記
- Fable 復帰時にレビューできるよう、判断の**根拠**（なぜ独立にしたか・なぜ折ったか）を背骨に残しておく

## 4. 背骨テンプレ（1 系列分・コピペ用）

```markdown
## 系列N：〔単元名〕

- **id 案**: `xxx_yyy_01`
- **中心の問い（胚細胞）**：「…はなぜ？——…だとしたら？」
  - 胚細胞的概念：**〔背後の転移可能な原理を 1 句で〕**。転移先：〔単元を超えて効く場面〕
- **既知→未知の橋**：系列N−1（…）＋ `[用語]`（…ユニット既習）。step1 Socratic 案：「…？」

| step | 内容（構造・値は実装時にオリジナル） | op | 答えの型 / affordances | 図 |
|---|---|---|---|---|
| 1 | …（既知からの入口） | 基本/Socratic | `[sqrt]` 等 | ● 図の内容（新規/流用） |
| 2〜10 | …（一度に差異 1 つ） | 同/逆/＋α/質的変化/複合 | … | 質的変化 step のみ ● |

- 5 オペレータ：同(…)・逆(…)・＋α(…)・質的変化(…)・複合(…) ✓
- **辞書**：`〔用語〕`（新規/更新）。`〔既存用語〕` と双方向
```

## 5. 役割の整理（このループの分担）

| 役割 | 担当 |
|---|---|
| 素材の消化（消化メモ・vault 非公開） | Opus |
| 背骨設計（本手順・docs/ 公開可） | 本来 Fable。**不在時は Opus が本文書で代行** |
| 背骨の検証・凍結の裁定 | 岩井先生（紙の本の実物と歩行感覚が最終裁定） |
| 実装（値起こし・SymPy 検証・ヒント 3 層・辞書・図 SVG） | Opus（既済背骨の実装事項に従う） |
| 実装レビュー・背骨の監査 | Fable（復帰時）。不在が続く間はメインが自己監査＋先生検証で代替 |
