# 推理式指導読方MVP仕様書（国語版第3弾）

> 数学版第3弾は動的生成・バックエンド前提で書かれ、実装フェーズで「手書き系列・静的サイト・localStorage」へ健全にピボットした（第4弾）。国語版第3弾は**その授業料を最初から払い込んだ仕様書**である——手書きが正、静的が正、実行時LLMなし、audit は系列第1号と同時。そのうえで国語固有の三つの新しい部品を設計する：**モーラ入力系（C1）・観点リストの版管理・句会支援画面**。MVPは俳句。17音の定型が、機械に支えられる入力系と、パブリックドメインの模範句と、句会という閉じの装置を同時にくれる。

**位置づけ**：国語版第3弾の構想文書。前2弾の延長線上の最終設計層。実コードは書かず、系列の背骨・データモデル・入力系・UI・audit・実装順序を詰める。**数学版第3弾（[[推理式指導算術MVP仕様書（第3弾）]]）の形式を鏡としつつ、内容は既存 `ruisuishiki` リポジトリ（commit e0bb341 時点の構造）への加法的増築として書く。**

## 0. 実装方針（2026-07-02 裁定済み事項の集約）

第2弾§8・§13 の裁定を実装方針として固定する：

- **同一リポジトリ・同一サイト・別ユニット**（Q3裁定）。`ruisuishiki` に国語ユニットを増築する。URL 名前空間・カタログ・audit 規則ファイルは教科別、UI基盤・型・履歴・デプロイは共有
- **MVPは俳句ユニット**（Q2裁定）。ブックレビューは第二陣
- **手書き系列が正・実行時LLMなし・静的エクスポート＋localStorage を維持**（第2弾§8.2）。動的生成もオンライン機能も作らない
- **入力系ゲート（C1）を最初に**：モーラ（音数）カウンタと産出ステップ型の実装・テストが段階1。これが通らない限り系列実装に入らない（数学版が√で払った授業料）
- **audit_kokugo は系列第1号と同時に作る**（第2弾§8.1。ヒントdriftの予防を後回しにしない）
- **句会はアプリ外の運用＋アプリは支援画面まで**（Q4裁定）。オンライン句会は共有可否フラグで道だけ残す
- **背骨（本書・docs/）に模範句の現物を書かない**（第2弾§8.5。著作権G12と公開可能性の両立）。本書は要件だけ書き、現物は実装時に調達・自作する
- **実装は Opus 系セッションへ委譲可能な粒度で書く**（memory `backbone_design_handoff` の分担表どおり：背骨＝本書、実装＝Opus、裁定＝先生、レビュー＝Fable）

## 2. 全体マップの中でのMVP位置

第2弾§7（三層）×§8.6（ジャンルの帯）で国語版の全体マップを描くと：

```
第3層（文章＝ジャンルの帯）: 俳句短歌 → ブックレビュー → ショートフィクション → 説明文 → …
第2層（文）              : 文をつくろう（小1）→ 接続語 → 主述修飾 → …
第1層（語）              : 四要素（語意・類義語・対義語・語法）→ 短文作成 → …
                            ※横軸は教科書単元配列・上限は学習指導要領
```

**MVP＝第3層の最初の帯「俳句」**。ただし俳句は三層の交点として例外的に有利な位置にある：17音の定型は**文章でありながらほぼ文のサイズ**であり（第2層への橋）、季語・オノマトペ・切れ字は**語の学習そのもの**である（第1層への橋）。俳句ユニットを歩くことが、三層構造の最小の実証になる。

**MVPゴール**：「学習者（まず先生自身と学級・家庭）が俳句系列を歩き、読み比べ→観点の発見→本歌取→自作→句会（運用）が一巡し、履歴がジャンル×オペレータ×観点で残る」状態。

## 3. MVP範囲の確定——俳句ユニット

### 3.1 選択理由（第2弾§13.1の再掲＋具体化）

- **C1（入力系）**：音数・季語・切れ字の検査は決定的に機械化できる。国語で唯一「SymPy的な柵」に近いものが立つジャンル
- **著作権（G12）**：芭蕉・蕪村・一茶・子規・漱石・虚子（1959没・2030年PD化までは要注意→**戦前没の俳人に限定**すれば安全）の名句群がパブリックドメイン。模範句調達が最も軽い
- **実践資産**：岩井2017の俳句実践（アンソロジー130句超・句会の運用・本歌取・「よい俳句の特徴」リスト）がそのまま設計の種になる
- **閉じの装置**：句会という文化装置が確立済み（Q4）
- **教科書並走（Q1）**：小学校国語教科書は各学年に俳句・季節の言葉の単元を持つ。担任学年の単元と並走できる

### 3.2 俳句ユニットの系列構成（背骨レベル・3系列）

模範句の現物と具体的ヒント文面は書かない（§0）。各系列は 8〜10 step、5オペレータ網羅、牧口四段階（読み比べ→観点抽出→制約付き産出→自由制作）を step 列に乗せる。

#### 系列1「五七五のかたち（音数とリズム）」
- **中心の問い（胚細胞）の要件**：「なぜ俳句は17音で足りるのか／17音という器が何をしてくれるのか」の方向で、実装時に memory `driving-question-embryo-cell` の checklist を通して1文に絞る
- **step 骨格**：
  1. 読み比べ（模範句2句の並置。音数を**声に出して**数える——経験から出発）〔基本原形〕
  2. 音数の識別（拗音・促音・長音・撥音を含む句で数える）〔同〕
  3. 語順の比較（同じ語群の並べ替えで印象がどう変わるか——並べ替え入力）〔逆：上五と下五の交換〕
  4. 字余りの句と定型の句の比較（外れも俳句であることを知る）〔＋α→観点の相対化〕
  5. 本歌取・一語差し替え（穴埋め産出。音数メーターが支える）〔同・最小の産出〕
  6. 二語差し替え〔同→＋α〕
  7. 気づきの言語化（この系列で見つけた「よい俳句の特徴」を選ぶ・書く）〔観点抽出〕
  8. 一句自作（題材は自由＝choiceは題材に。音数メーター常駐）〔自由制作〕
- **判定の性格**：step2〜4 は決定的判定（音数・選択）。step5〜8 は**制約充足＋自己チェック**（§6.3）

#### 系列2「季語——季節のことばが景色をはこぶ」
- **中心の問いの要件**：「たった一語の季語が、なぜ句全体の景色を変えるのか」の方向
- **step 骨格**：季語の識別（選択）→同じ句型で季語だけ替えた2句の比較（**質的変化の核**：季語の交換で同じ光景が別の季節・別の感情になる）→季語から連想を広げる（語の系列への橋：季語の類義語・連想語）→季語を選んで本歌取→季語から一句
- **オペレータの重点**：質的変化（季語交換）・逆（句から季節を当てる⇄季節から季語を選ぶ）

#### 系列3「切れ——『や・かな・けり』が向かい合わせるもの」
- **中心の問いの要件**：「切れ字は何と何を向かい合わせているのか」の方向（対比＝戸田の対義語・テーゼ⇔アンチテーゼ構造への橋）
- **step 骨格**：切れ字の識別→切れの前後で何が並置されているかの比較（対比の指さし）→切れ字の有無で同想の句を読み比べ（＋α）→切れ字を使って本歌取→一句
- **オペレータの重点**：＋α（切れ字という制約の追加）・複合（季語＋切れ字＋対比）

**縦の鎖**：系列1（器）→系列2（景色を運ぶ語）→系列3（向かい合わせる構造）。辞書（§5.6）の `relatedSeriesId` で双方向に連動（数学版の三角ユニットと同じ設計）。

### 3.3 国語版5オペレータの俳句での運用（第2弾§5.2の具体化）

| Op | 俳句での実装 |
|---|---|
| 同 | 本歌取（一語差し替え→二語→句型だけ借りる、の勾配）。同じ句型で別の題材 |
| 逆 | 読み⇄書きの交換（読んだ型で詠む）。語順交換（上五⇄下五）。句→季節を当てる⇄季節→季語を選ぶ |
| ＋α | 制約の追加：切れ字を使う・オノマトペを入れる・数字を入れる・色の語を入れる（岩井の「よい俳句の特徴」リストの各項目が制約在庫） |
| 質的変化 | 季語の交換（同じ光景が別の季節になる）。俳句→短歌（構造保存の拡張・第二陣）。川柳（季語なし）との比較 |
| 複合 | 上記の組み合わせ（例：季語を替えて・切れ字を足して・別の題材で） |

**配列制約：一度に差異点は一つ**（数学版と同一。系列内の連続 step で二段飛ばしをしない）。

### 3.4 利用者像と利用シナリオ

- **利用者**：まず岩井先生自身（学習駆動——数学版と同じ開発サイクル）と家庭・学級。学年は可変設計（ふりがな・語彙は小3〜4を既定に、設定で上下）
- **シナリオA（学習者・15分）**：系列を開く→模範句の読み比べ→気づきを選ぶ／書く→本歌取→音数メーターを見ながら自作→清書カードに表示→（教室・家庭で）読み合う
- **シナリオB（句会・教室運用）**：各自の清書カードを見せ合う（黒板・大型テレビ・紙）→特選・お気に入りの選（岩井方式：特選2点・お気に入り1点）→最高得点句の「なぜよかったか」を話し合う→**アプリの句会記録欄に観点を記録**→観点リストに追記（版が上がる）
- **シナリオC（教師・5分）**：履歴ビューで「どのオペレータ・どの観点でヒントを開いたか」を見る→次に歩く系列・補充を判断

## 5. データモデル詳細（types.ts への加法的拡張）

TypeScript の型定義レベルで示す。**既存フィールドの変更はゼロ**、すべて追加。

```typescript
// 5.1 step の種別（新規・国語ステップの根幹）
// 既存の数学 step は kind 未指定＝ "exercise" として後方互換
type StepKind =
  | "exercise"     // 判定あり（音数識別・季語選択・並べ替え等。決定的判定）
  | "comparison"   // 読み比べ（模範句の並置＋気づきの選択/記述。正誤なし）
  | "creation";    // 産出（本歌取・自作。制約充足＋自己チェック。正誤なし）

// 5.2 国語系 step の入力仕様（inputAffordances の拡張として）
type KokugoInput =
  | { type: "choice"; options: string[]; answerIndex: number }        // 決定的判定
  | { type: "reorder"; segments: string[]; answerOrder: number[] }    // 並べ替え
  | { type: "fillIn"; template: string; slotConstraints: SlotConstraint[] } // 本歌取
  | { type: "haikuText" };  // 自由律の1行入力＋よみがな欄（§6.2）

type SlotConstraint = {
  moraCount?: number;          // このスロットは5音、等
  mustBeKigo?: boolean;        // 季語であること（季語表と照合）
  forbidWords?: string[];      // 元句の語（丸写し防止でなく「差し替え」の担保）
};

// 5.3 産出 step の充足チェック（判定ではなく可視化・G2）
type CreationCheck = {
  meterTarget: number[];       // [5,7,5]
  meterPolicy: "visualize";    // 正誤にしない。ぴったり/字余り/字足らずを表示するだけ
  kigoCheck?: "visualize";     // 季語表との照合結果を表示するだけ
  selfChecklist: string[];     // 観点リストからの自己チェック項目（読み比べ後にのみ表示＝G1）
};

// 5.4 模範文（MentorText）
type MentorText = {
  id: string;
  text: string;                // 句の本文
  reading?: string;            // よみがな（音数表示用）
  author: string;
  sourceNote: string;          // 出典（memory ruisuishiki-citation-format の書式）
  rights: "PD" | "original" | "licensed";  // G12。PD は戦前没の俳人に限定
  kigo?: { word: string; season: "春" | "夏" | "秋" | "冬" | "新年" };
  viewpointTags: string[];     // この句が担う観点（"オノマトペ" "対比" 等）——教師/設計者ビュー用。子どもUIには出さない（G1）
  shareable: boolean;          // 学習者作品を模範文プールに入れる場合の許諾フラグ（将来のオンライン句会の道もこのフラグ）
};

// 5.5 観点リスト（ViewpointList）——版を持つ生き物（第2弾§3.3）
type ViewpointList = {
  genreId: string;             // "haiku"
  version: number;
  items: ViewpointItem[];
};
type ViewpointItem = {
  text: string;                // "季語がある" "音をそろえている（押韻）" 等
  addedIn: "initial" | "kukai" | "reading";  // 句会由来の追記が記録される（Q4裁定の核）
  addedAt?: string;            // ISO日付
};

// 5.6 句会記録（KukaiRecord）——localStorage。アプリは記録係（Q4）
type KukaiRecord = {
  date: string;
  seriesId?: string;
  bestWorkNote?: string;       // 最高得点句のメモ（本文は書かなくてもよい——個人情報最小化）
  whyGoodNotes: string[];      // 「なぜよかったか」の観点メモ → ViewpointList への追記候補
};

// 5.7 履歴（StepRecord の拡張——集計軸の追加）
// 既存 StepRecord に加法的に：
//   viewpointRefs?: string[]   … その step で参照/自己チェックした観点
// 集計ビューの軸：ジャンル × オペレータ × 観点 × ヒント到達層（G8。表示層まで§7.4で設計）
```

### 5.8 設計上の重要ポイント

- **creation step に「模範解答」フィールドを存在させない**（代筆禁止の型レベル実装。audit でも検査——§8）
- **meterPolicy: "visualize"**——音数チェックを正誤にしない。「字余りも俳句」（岩井2017：約束事があるから外れを学べる）。型が学習者を裁かず、情報を返す
- **観点タグは MentorText 側に持ち、子どもUIに出さない**——観点は読み比べから発見される（G1）。教師・設計者ビューでのみ見える（数学版の文型タグと同じ扱い＝F1）
- **ViewpointItem.addedIn: "kukai"**——句会の「なぜよかったか」が観点リストの版を上げる。閉じの装置とエンジンがデータで接続する（Q4裁定の実装）

## 6. 入力系の仕様（C1ゲート・段階1）

### 6.1 モーラカウンタ（moraCount.ts）——決定的・純関数

- 入力：ひらがな/カタカナ文字列（よみがな）。出力：モーラ数と分節列
- 規則：
  - 拗音（ゃ・ゅ・ょ・ぁ行の小書き）＝直前の仮名と合わせて1モーラ（「きゃ」=1）
  - 促音「っ」＝1モーラ／撥音「ん」＝1モーラ／長音「ー」＝1モーラ
  - カタカナ・全角半角・濁点合成文字は正規化（NFC）してから数える
  - 空白・句読点・記号は数えない
- **数えないもの（仕様の輪郭として明記）**：漢字かな交じり文の自動読み推定はしない（形態素解析を持ち込まない——静的サイトの軽さを守る）。よみがな欄が正
- テスト（`test_mora_count.ts`・数学版 `test_answer_eval.ts` 方式で収録）：拗音・促音・長音・撥音・小書き・カタカナ・全半角混在・記号混入・空文字列、の各ケース。**実装より先にテストケース一覧を書く**（数学版 T-2 の教訓：テストは監査で作られ、収録されて初めて資産になる）

### 6.2 俳句入力の2欄方式

- **作品欄**（漢字かな交じり可・自由）＋**よみがな欄**（ひらがな。音数メーターの入力）
- 低学年設定では作品欄＝よみがな欄を兼ねる（ひらがな1欄）
- 音数メーターは入力中リアルタイムに「○○○○○｜○○○○○○○｜○○○○○」形式で分節表示。5-7-5 ぴったり・字余り・字足らずを**色ではなく形で**も区別（アクセシビリティ）

### 6.3 「判定」と「可視化」の区別（G2の実装原則）

| step 種別 | 機械がすること | 機械がしないこと |
|---|---|---|
| exercise（識別・並べ替え） | 正誤判定（決定的） | — |
| comparison（読み比べ） | 気づきの記録 | 気づきの採点 |
| creation（本歌取・自作） | 制約の可視化（音数・季語照合・スロット充足）＋観点セルフチェックの提示 | 作品の合否宣告・点数化・添削・代筆 |

## 8. 比較の指さしの規律と audit_kokugo（段階1〜2と同時）

### 8.1 手書きヒントの規律（実行時LLMなし——drift は執筆時に起きる）

数学版の教訓：drift は「親切な執筆者」のデフォルト引力で起きる（手書き系列の約8割が違反した）。国語版の3層：

| Layer | 書いてよいこと | 書いてはいけないこと |
|---|---|---|
| L1 | 「（模範句カードを開いて）前の句と比べてみよう。何が同じ？何が違う？」 | 観点の名指し・技法名 |
| L2 | 差異/観点**1つ**を問いの形で（「この句の季節のことば、どれだろう？」「切れ字の前とあと、何が向かい合ってる？」） | 「切れ字を使うと感動が強まります」等の解説文・「〜と書きましょう」 |
| L3 | **模範句側**の該当箇所の並置＋観点の言語化（「この句では『や』の前に◯◯、あとに△△が置かれている」） | 学習者の句の完成形・改作例・穴埋めの答えの例示 |

### 8.2 audit_hints_kokugo.py の検出規則（報告専用・CI化しない——development-loop の方針どおり）

1. **L1/L2 の禁止パターン**：技法名の先出し（「切れ字」「季語」「オノマトペ」等の語が L1 に出る／L2 で解説調「〜です・〜ます＋効果の説明」）、指示調（「〜と書きましょう」「〜を使いましょう」）
2. **L3 の代筆検出**：creation step の L3 に完成句形式（5-7-5 のモーラ列）が含まれていないか（モーラカウンタを流用して機械検出——国語版ならではの検査）
3. **データモデル検査**：creation step に模範解答フィールドが存在しないか／観点セルフチェックが読み比べ step より前に出ていないか（G1）
4. **オペレータ網羅**：5オペレータ最低1 step（数学版の検査を流用）
5. **出典検査**：MentorText 全件に sourceNote と rights があるか（G12）

### 8.3 お手本基準（国語版7条件）の作り方

数学版の「お手本7条件」は**お手本2系列の比較から抽出**された（比較が基準を生む）。国語版も同じ手順を踏む：**系列1・系列2 を実装→両者を比較して国語版お手本条件を明文化→系列3 と以後の系列はそれを mirror**。本書では条件を先取りで確定しない（先取りは G1 の設計者版——基準は比較から生まれるべき）。候補観点だけ挙げておく：①中心の問い（胚細胞）②5オペレータ網羅 ③ヒント3層（L2＝差異1つ）④読み比べ→観点→産出の牧口勾配 ⑤辞書連動 ⑥模範句の出典・権利 ⑦音数メーター等の足場のフェードアウト設計。
