## 3. audit_hints_kokugo.py の既知の限界（2026-07-03 較正済み）

audit が緑でも以下は**見ていない**。T-3（第6弾タスク計画）で収録するまでは人力で守る：

1. **漢字まじりの完成句**（代筆検出は「かな連続12音以上」のみ。代筆の主形態を検出できない）
2. **指示調の亜種**（「〜を入れましょう」「選びましょう」等）
3. L1 検査の感度（「かな」「どこ」等を含めばほぼ通る——L1 は目視も併用）
4. **系列単位の判定**（オペレータ網羅・selfChecklist 順序が3系列合算。ファイルも `seriesKokugoHaiku.ts` ハードコード）
5. mentorTextRefs 整合・reading 音数検算・未使用句報告（audit に無い）

偽陽性は確認済みでゼロ（問題文・drivingQuestion の技法名は検査対象外。「きれい」は除外される）。

## 4. 「Fable 級の判断が本当に必要な場面」の見分け

**必要なのは4場面だけ**（それ以外は Opus で足りると宣言する）：

1. **基準を生む瞬間**——新ジャンル（ブックレビュー・文の系列）の最初の系列群を比較して「お手本条件」自体を作り直すとき。※短歌は俳句の7条件を mirror できるので Opus で足りる
2. **正典と実装が二股に割れ、どの条項を優先するか自体が争点**のとき（例：NFC→NFKC のような正典側の誤り認定）
3. **中核ふるまいの境界事例**——「これは代筆か足場か」「基準の先出しか土俵づくりか」が文言レベルで割れるとき
4. **失敗モードの新種（G13 候補）**を疑うとき

**迷ったときの判定式**：「この判断を間違えたとき、気づける機械（audit・テスト）か文書（7条件・G1〜G12）が既にあるか？」——あるなら Opus で進めて還流で拾う。ないなら先生が止めて二股裁定リストに積み、第6弾 Open Questions の形式（**推奨＋根拠＋逆を採る条件**）で先生自身が裁定を書き残す。
