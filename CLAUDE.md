# CLAUDE.md — ruisuishikiプロジェクト

## このプロジェクトについて

戸田城外『推理式指導算術』（1930）の方法論をAI時代に再構成するMVPの実装リポジトリ。

最終ゴールは、子どもも大人も学べ、教師も大人も自分で教材を生み出せる**世界最高峰の算数・数学教育プラットホーム**。

「教育」は「学習」と「教材生成（指導）」の両方を含む——第1弾§5.3の「同じエンジンの二つのインターフェース」原則を旗印として日本語で表現したもの。実装上、教師ビューと学習者ビューが同一の生成系から派生する設計を必ず維持する。

## ビジョン声明（実装の判断軸の上位原則）

以下はコードレベルの判断にも波及する上位原則。設計のトレードオフで迷ったら、この声明に立ち返って判断する：

- **無料**で公開する——ビジネスモデルや価格競争を超えて、誰でも触れられること自体を価値とする
- **GitHub Pages で公開**——インフラ依存を最小化し、メンテナンスが続く限り誰でもアクセスできる。フロントエンドは static export 前提で設計
- **デザイン・UI/UX も世界最高を目指す**——機能だけでなく、触れた瞬間の体験そのものを設計する
- **文理を問わず、入口を狭めない**——文系の大人、苦手意識のある子ども、得意な子・先取りしたい子、誰でも入れる
- **陳腐にならない**——既製のドリル・解説サイトとは違う、戸田の推理式設計を貫いた天下無双の質
- **初心者に入りやすく、それでいて陳腐でない**——ローフロア（誰でも入れる）と高い天井（数学者まで使える）を両立する

### このビジョンが実装に与える具体的な含意

1. **アーキテクチャ**：フロントエンドは **静的サイトで配布可能**な構成を保つ（Next.js の static export または同等）。バックエンド（SymPy、LLM呼び出し）は別ホスト（Cloudflare Workers / Render / Railway 等の無料枠で運用可能なもの）に分離
2. **依存とランニングコスト**：常時稼働する有料サービスを増やさない。LLM 呼び出しは「教師の生成時のみ」に集約し、学習者ビューは生成済み問題プールを静的に配信する設計を優先
3. **UI/UX**：第3弾§7のワイヤーフレームは出発点に過ぎない。**触れた瞬間の体験**を最重視し、視覚言語・タイポグラフィ・動きの設計を妥協しない
4. **アクセシビリティ**：文理問わず・初心者が入れる——タブレットでもデスクトップでもキーボード操作でも使える設計、特別支援にも配慮した色彩・余白・フォント選択
5. **オープンソース化**：MVPが安定したら公開ライセンス（MIT or Apache 2.0）で公開。コントリビューションの受け入れ準備も視野に
6. **データプライバシー**：無料公開でも個人情報は最小収集に留める。学習履歴はローカルファースト（ブラウザストレージ優先）も検討

## 中核原則（実装の判断軸）

1. **文型スキーマと5つの変化オペレータ（同・逆・＋α・質的変化・複合）が一級市民**——データモデルでも、生成パイプラインでも、診断軸でも
2. **「比較を指さすAI」を絶対に裏切らない**——解説・公式名・部分計算・答えの示唆をLLMの応答に含めない。プロンプト＋後処理フィルタで二重に防ぐ
3. **フェードイン解説3層**——初学者向けworked exampleは「学習者が選んで開示する」第3層に隔離する
4. **集計軸は「文型×オペレータ」**——「正答率」だけで履歴を持たない（第2弾失敗モードF8の予防）
5. **解の一意性が破れる文型は問題プールから除外**——「答のみで自得」の前提を崩さないため、SymPyの一意性検証は必須
6. **子ども側UIに文型タグを出さない**（F1の予防）——文型は教師ビューでのみ可視
7. **教師の専門性を萎縮させない**——AIは「提案」、最終判断は教師（F7の予防）

6. **図**：Step 1 と質的変化 Step のみ・答えを書かない
7. **出典**：APA・構成は借り値はオリジナル（memory `copyright-credit-vs-copy`）

**追補：経験の質の基準 Q1〜Q7**（2026-07-17・Fable 5 が三角続き 6 系列の実装から抽出。7 条件が骨格＝機械チェック可能な必要条件、Q1〜Q7 は「歩いたときに何が起きるか」＝検収時の観点リスト）。正は `docs/quality_bar_fable_2026-07-17.md`：

### ヒント 3 層の必須構造（絶対譲らない）

各 step の `hints` は **必ず** 次の 3 層構造を持つ：

| Layer | 役割 | 書き方 |
|---|---|---|
| **Layer 1** | **比較指さし呼びかけ** のみ | 「**前題と比べてみよう。何が同じで、何が違う？**」のような問いかけ。具体的な手順・公式・計算式を **書かない** |
| **Layer 2** | **差異を「1 つ」名指しする** | 「前題と変わったのは○○だけ」と、変わった 1 点だけを名指し（または問いかけ）。Layer 1 の言い換えにしない。「どこが変わったか」を 1 つに絞るのが Layer 2 固有の仕事。解法・計算は言わない |
| **Layer 3** | **完全な worked example（最終手段）** | 前題の解法を示し、現問題との差を 1 つ明示。**式・計算・答えを出してよい**（最終手段なので確実に立たせる）。完全さは難度に応じてスケール（易しい同オペレータ step は簡潔に／質的転換 step は文字式を見せてしっかり）。砦は Layer 1・2。**ヒストリーに出していない式を指す「宙ぶらりん参照」を作らない**——必要な式は使う step（または導入 step）の Layer 3 で見せる |

**Layer 1 / Layer 2 で絶対に書いてはいけないこと**（第 3 弾 § 6.3.2 の禁止条項）：

- ❌ 現問題の解き方を直接示すこと（「$t = x + 1$ を代入する」「両辺を 2 乗する」）
- ❌ 答えを示すこと（Layer 3 まで温存）
- ❌ 公式名を直接出して当てはめさせること（「解の公式を使う」）
- ❌ 部分計算を出すこと（「$y = 2(x+1)+5$」）

**比較指さしの語彙パターン**：「**前題と比べると**」「**何が同じ？ 何が違う？**」「**前題で効いた方針が、今はどう効く？**」「**Step N で身につけた手順はそのまま使える？**」

### Step 1（first）の特例

最初の step には「前題」が無いので、Layer 1 は比較指さしの代わりに **Socratic な問いかけ**（ソクラテスの問答＝答えを教えず問いで気づかせる。[[産婆役としての教師]]）を置く：「○○を式にするとどうなる？」「○○の関係を使うと何が見える？」

**step1 は既知（前学年・前単元・生活の概念）から問いで導く（既知→未知の橋）**：
- その問題を解くのに要る土台（割合・平均・へだたり 等）を手がかりに問う。既存の「その**割合**を考えてみよう」「**へだたり**を数で言うと」が良い例
- その既知の鍵概念を **`[用語]` でリンク**し、忘れた子が辞書 → 土台の系列へ**学び直せる**ようにする（低い床。持っていない子の救済）
- `relatedSeriesId` は概念の **最も下＝最初に教わる系列** を指す。こうすると辞書が「既知→未知」の背骨になる（例：割合→`ratio_basic_01`、平均→`e5_mean_01`、中点→`adv_number_line_01`）
- 同じ概念が後続系列で再び出る所では「○○でやった△△」と前系列を指して**縦の鎖**をつなぐ（例：直線の中点＝数直線の平均）。ただし**強制せず、自然に効く入口だけ**。前提系列がまだ無ければ `[用語]` は素通しで degrade（前提系列ができたら live になる）→ だから**設計は今・リンクは前提系列ができるたび**でよい

### 5 オペレータすべてを最低 1 問ずつ含める

10 step の系列なら、{ 同 (SAME) ・ 逆 (INVERSE) ・ ＋α (PLUS_ALPHA) ・ 質的変化 (QUALITATIVE) ・ 複合 (COMPOSITE) } をすべて含める。**特に逆 (INVERSE) と ＋α (PLUS_ALPHA) を意識**——私の癖でこれらが抜けやすい。`python3 scripts/audit_hints.py` がオペレータ網羅も検査する（不足は「⚠オペレータ欠落」で表示）。コミット前に ⚠ が出ないこと。逆は「答→問題」（中点や内分の結果から端を逆算）、複合は質的変化＋逆などの重ね技で自然に作れる。

### 中心の問い（drivingQuestion）は胚細胞モデル

中心の問いは **「単元の胚細胞的概念（背後の転移可能な原理）」を 1 文の問いに絞る**。さらに **その単元自身の胚細胞に接続**させる（図形と方程式なら「図形 ↔ 座標・式の翻訳」）。
- ❌ 扱う操作・話題の列挙（「距離・中点・分ける点を計算で出せないか？」「傾き・切片・中点・垂直をどう読む？」）
- ❌ 観察可能な手続きや結果を問う（「どんな図形が描かれる？」「解の公式を使うには？」）
- ✅ 背後の 1 原理を問い、単元の胚細胞に接続する。例：
  - 数直線上の点「直線上の点の位置は、座標の『差』と『重み付き平均』だけで——定規を当てずに——どこまで言い当てられる？」
  - 直線の方程式「たった 1 本の式 $y=mx+n$ が平面の直線を完全に決めてしまうのはなぜ？ 図形と式は、どこまで翻訳しあえる？」

**基準の出し方**：お手本 2 系列の中心の問いを **比較**すると、不変項＝「操作の列挙でなく、背後の 1 原理を、単元の胚細胞に接続して問う」が立ち上がる。これが基準（比較が基準を生む＝推理式の方法のメタ適用）。

詳細：memory `feedback_driving_question_embryo_cell.md`

### 図（figureMarker）の配置

### 国語版お手本系列の品質基準（7条件・コミット前に全部 ✓）

俳句3系列（五七五・季語・切れ）の**比較**から抽出した不変項（2026-07-03 国語版第6弾。比較が基準を生む＝数学版7条件と同じ手順）。新規系列（短歌・ブックレビュー・文の系列）はこれを mirror する：

1. **中心の問い＝胚細胞**：共通胚細胞「小さな装置（型・語・配置）が大きな意味の体系を呼び出す」の一つの相を「なぜ」の1文で問う。この系列がどの装置の相かを先に言えること
2. **5オペレータ網羅（系列単位）**＋質的変化は系列の核の発見と一致させる。差異は一度に1つ
3. **ヒント3層＝比較の指さしの国語訳＋言い換えの規律**：L1 技法名ゼロの呼びかけ／L2 差異1つ・技法名は子どもの語彙に言い換え（季語→「季節のことば」、切れ字→「区切る字」）／L3 **模範句側**の言語化（技法名・[用語] 解禁）。**代筆は L3 でも絶対なし**（「作り方は元の句が見せている」で終わる）
4. **牧口勾配の step 骨格**：読み比べ（声に出す）→識別→最小対の比較→本歌取（一点→幅拡大）→観点抽出（pickViewpoints・読み比べの後・核観点は revealedInSeries で解禁）→複合→自由制作（清書カード→句会へ）
5. **最小対の比較ペア＋調達規律**：比較ペアは一点差の相方（相方は自作が主力）。模範句は全件 sourceNote＋rights（PD＝戦前没・没年明記）・reading を countMora で検算・ユニーク7〜9句
6. **足場のフェードアウト**：図は step1＋質的変化のみ／メーターは可視化のみ（字余りも俳句・判定しない）／観点リストは薄く生まれ系列の核は解禁で増える／俳句本文は縦書き1列 nowrap
7. **辞書連動と縦の鎖**：[用語] リンク＋relatedSeriesId 双方向・後続系列は前系列を名指す。L1/L2 には技法名も [用語] も出さない

### 開発フェーズ（MVP段階1〜6）
**1+3ハイブリッド戦略**で進める。Anthropic API課金は当面行わない。

- 実装上は `generation` モジュールに **mock モード** を持たせ、JSON プールから読み込む形にする
- 将来API課金が必要になったら、同じインターフェースで live モードに切り替えられる設計

### 運用フェーズ（本番公開後）

| 役割 | コスト | 仕組み |
|---|---|---|
| 学習者（子ども・大人） | **完全無料** | GitHub Pages から静的データを消費するだけ。LLM呼び出しなし |
| 教師（教材生成したい人） | **BYOK**（自分のAPIキー） | 設定画面に Anthropic API キーを入力し、自分のアカウントに課金される形 |
| プラットホーム運営 | **ゼロ** | GitHub Pages（無料）＋静的データ配信のみ |

### 実装上の含意

- 学習者ビューは API キーを一切要求しない・LLMを呼ばない（静的データのみ）
- 学習履歴は **ブラウザのlocalStorageに保存**（運営にデータが渡らない・プライバシー保護）
- 教師ビューは BYOK 入力欄を持つ。キーはブラウザに保存され、運営サーバーには送らない
- 「生成済みプール」と「教師がBYOKで生成した新規」を共存させる UI
- データプライバシー：学習履歴・作問・教師作成の系列、すべて運営サーバーに保存しない設計を優先

これらは GitHub Pages 無料公開・天下無双の質・初心者にも入りやすいビジョンと整合する。

## ゲーミフィケーション方針（将来計画・2026-06-14 確定）

将来的にゲーミフィケーション要素を導入する。ただし**内在的**と**外付け**を厳密に区別する。詳細は `docs/design_system.md` の「ゲーミフィケーション設計の指針」を参照。

**要約**：
- **内在的ゲーミフィケーション**（戸田の推理式設計に内在）→ 積極的に実装。質的変化の発見の喜び・系列の達成・作問が採用される実績・つまずきオペレータの解消軌跡
- **外付けゲーミフィケーション**（バッジ・ポイント等）→ 慎重に・最小限・上品に。原則ポイント不使用、SNS共有を煽らない、キャラクター導入しない
- **判断軸**：「戸田の哲学（自分で考えること自体に価値）を強化するか、外発的動機に置き換えるか」

実装タイミング：段階7（拡張フェーズ）以降。MVPでは内在的な体験設計（visualの温度感・アニメーション）を仕込んでおく。
