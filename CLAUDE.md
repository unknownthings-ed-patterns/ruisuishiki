# CLAUDE.md — ruisuishikiプロジェクト

## このプロジェクトについて

戸田城外『推理式指導算術』（1930）の方法論をAI時代に再構成するMVPの実装リポジトリ。

最終ゴールは、子どもも大人も学べ、教師も大人も自分で教材を生み出せる**世界最高峰の算数・数学教育プラットホーム**。

「教育」は「学習」と「教材生成（指導）」の両方を含む——第1弾§5.3の「同じエンジンの二つのインターフェース」原則を旗印として日本語で表現したもの。実装上、教師ビューと学習者ビューが同一の生成系から派生する設計を必ず維持する。

## ビジョン声明（実装の判断軸の上位原則）

以下はコードレベルの判断にも波及する上位原則。設計のトレードオフで迷ったら、この声明に立ち返って判断する：

- **無料**で公開する——ビジネスモデルや価格競争を超えて、誰でも触れられること自体を価値とする
- **GitHub Pages で公開**——インフラ依存を最小化し、メンテナンスが続く限り誰でもアクセスできる。フロントエンドは static export 前提で設計
- **デザイン・UI/UX も世界最高を目指す**——機能だけでなく、触れた瞬間の体験そのものを設計する
- **文理を問わず、入口を狭めない**——文系の大人、苦手意識のある子ども、得意な子・先取りしたい子、誰でも入れる
- **陳腐にならない**——既製のドリル・解説サイトとは違う、戸田の推理式設計を貫いた天下無双の質
- **初心者に入りやすく、それでいて陳腐でない**——ローフロア（誰でも入れる）と高い天井（数学者まで使える）を両立する

### このビジョンが実装に与える具体的な含意

1. **アーキテクチャ**：フロントエンドは **静的サイトで配布可能**な構成を保つ（Next.js の static export または同等）。バックエンド（SymPy、LLM呼び出し）は別ホスト（Cloudflare Workers / Render / Railway 等の無料枠で運用可能なもの）に分離
2. **依存とランニングコスト**：常時稼働する有料サービスを増やさない。LLM 呼び出しは「教師の生成時のみ」に集約し、学習者ビューは生成済み問題プールを静的に配信する設計を優先
3. **UI/UX**：第3弾§7のワイヤーフレームは出発点に過ぎない。**触れた瞬間の体験**を最重視し、視覚言語・タイポグラフィ・動きの設計を妥協しない
4. **アクセシビリティ**：文理問わず・初心者が入れる——タブレットでもデスクトップでもキーボード操作でも使える設計、特別支援にも配慮した色彩・余白・フォント選択
5. **オープンソース化**：MVPが安定したら公開ライセンス（MIT or Apache 2.0）で公開。コントリビューションの受け入れ準備も視野に
6. **データプライバシー**：無料公開でも個人情報は最小収集に留める。学習履歴はローカルファースト（ブラウザストレージ優先）も検討

## セッション開始時に必ず読むもの

実装の判断は、以下の文書群（Obsidian Vault `教育のパタン・ランゲージ/個人/`）の理論・設計仕様を**正**とする：

1. `推理式指導算術とAI時代の再構成（統合理論文書）.md`（第1弾・テーゼ）
2. `推理式指導算術AI構想の批判的検討（第2弾）.md`（第2弾・修正版テーゼ）
3. `推理式指導算術MVP仕様書（第3弾）.md`（第3弾・実装仕様書）

これら3文書とコードに齟齬があれば、**文書が正**。コードが文書と違うことをしているなら、コード修正か文書修正のどちらかを判断する必要がある。

## 中核原則（実装の判断軸）

1. **文型スキーマと5つの変化オペレータ（同・逆・＋α・質的変化・複合）が一級市民**——データモデルでも、生成パイプラインでも、診断軸でも
2. **「比較を指さすAI」を絶対に裏切らない**——解説・公式名・部分計算・答えの示唆をLLMの応答に含めない。プロンプト＋後処理フィルタで二重に防ぐ
3. **フェードイン解説3層**——初学者向けworked exampleは「学習者が選んで開示する」第3層に隔離する
4. **集計軸は「文型×オペレータ」**——「正答率」だけで履歴を持たない（第2弾失敗モードF8の予防）
5. **解の一意性が破れる文型は問題プールから除外**——「答のみで自得」の前提を崩さないため、SymPyの一意性検証は必須
6. **子ども側UIに文型タグを出さない**（F1の予防）——文型は教師ビューでのみ可視
7. **教師の専門性を萎縮させない**——AIは「提案」、最終判断は教師（F7の予防）

## 失敗モード監視（F1〜F10）

第2弾§9のチェックリストを実装側でも常に意識する：

- F1: 文型タグの早すぎる開示
- F2: 生成問題の解の不一意・整合性破綻
- F3: 表層特徴による抽象構造の発見阻害
- F4: 初学者にworked example不足で挫折
- F5: 「答のみ」が孤独な学習者に機能しない
- F6: 文型に乗らない探究的問題が排除される
- F7: AI依存による教師の専門性萎縮
- F8: 履歴が「正答率」に堕ちる
- F9: 受験産業への取り込み
- F10: 比較の過剰強制

## 実装方針

- **段階1〜6はClaude Code単独**で進める（文脈の一貫性を最重視）
- **Codex（GPT-5系）は拡張フェーズで投入**（段階7以降、複数学年展開・テスト自動化・CI整備）
- **段階6（フィールド試行）の前で必ず立ち止まる**——子どもに使わせる前にレビュー機会を挟む
- 「fable-5が復活したらレビュー」を想定し、コードと設計の対応関係を明示する

## 開発フェーズ・運用フェーズの課金モデル（2026-06-14 確定）

### 開発フェーズ（MVP段階1〜6）
**1+3ハイブリッド戦略**で進める。Anthropic API課金は当面行わない。
- **方法1（半手動）**：Claude Code チャット（Pro/Maxサブスク内）で対話的に材料を生成し、JSON ファイルに保存
- **方法3（生成済みプール配信）**：開発時に作った材料プールを `backend/data/material_pools/` に置く
- 実装上は `generation` モジュールに **mock モード** を持たせ、JSON プールから読み込む形にする
- 将来API課金が必要になったら、同じインターフェースで live モードに切り替えられる設計

### 運用フェーズ（本番公開後）

| 役割 | コスト | 仕組み |
|---|---|---|
| 学習者（子ども・大人） | **完全無料** | GitHub Pages から静的データを消費するだけ。LLM呼び出しなし |
| 教師（教材生成したい人） | **BYOK**（自分のAPIキー） | 設定画面に Anthropic API キーを入力し、自分のアカウントに課金される形 |
| プラットホーム運営 | **ゼロ** | GitHub Pages（無料）＋静的データ配信のみ |

### 実装上の含意

- 学習者ビューは API キーを一切要求しない・LLMを呼ばない（静的データのみ）
- 学習履歴は **ブラウザのlocalStorageに保存**（運営にデータが渡らない・プライバシー保護）
- 教師ビューは BYOK 入力欄を持つ。キーはブラウザに保存され、運営サーバーには送らない
- 「生成済みプール」と「教師がBYOKで生成した新規」を共存させる UI
- データプライバシー：学習履歴・作問・教師作成の系列、すべて運営サーバーに保存しない設計を優先

これらは GitHub Pages 無料公開・天下無双の質・初心者にも入りやすいビジョンと整合する。

## 技術スタック

- **段階1（CLI）**: Python 3.11+ / SymPy / Anthropic SDK
- **段階2以降（UI）**: Next.js + React + Tailwind / FastAPI / SQLite→PostgreSQL
- **記号計算はSymPy一択**——mathjs/nerdamerに切り替えない（世界最高峰の数学プラットホームの土台を10年使う前提）

## コーディング規約（暫定）

- Python: PEP 8、`ruff` でlint、`pytest` でテスト
- TypeScript: ESLint + Prettier標準、strict mode
- 関数・変数名は英語、ドメイン用語（文型・解式・変化オペレータ）はローマ字または日本語コメントで意図保持
- データモデル（Pattern, Material, Problem, VariationOperator, Series等）は第3弾§5の型定義を**正**

## ドメイン用語の翻訳表

| 日本語 | 英語コード | 備考 |
|---|---|---|
| 文型 | Pattern | 戸田の用語。問題の構造 |
| 材料 | Material | 数量・場面・文脈 |
| 関係 | Relation | 材料間の結びつき |
| 解式 | Formula | ＋−×÷で表現したもの |
| 変化オペレータ | VariationOperator | 系列の差分の型 |
| 同 / 逆 / ＋α / 質的変化 / 複合 | SAME / INVERSE / PLUS_ALPHA / QUALITATIVE / COMPOSITE | enumの値 |
| 比較を指さす | comparison pointing | ヒントの中核ふるまい |
| フェードイン解説 | fade-in explanation | 初学者向け足場の段階開示 |

## 安全規則

- **git config はローカル（リポジトリ単位）のみ設定**。グローバル `~/.gitconfig` は触らない
- **大きな変更を始める前にユーザーに確認**（実装方針の転換・スタック変更・データ構造の大改訂）
- **失敗モードF1〜F10に該当する設計はその場で警告**。修正案を提示してからコードを書く
- **コミットメッセージは設計文書のどの節を実装したかを明示**（例: 第3弾§5.3 P1パターン実装）
